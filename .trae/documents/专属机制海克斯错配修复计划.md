# 专属机制海克斯错配修复计划

## Summary

- 目标：复盘并修复 `Karma / Seraphine / Renata` 在 `Master of Duality / Spin To Win` 对比下出现 `Spin To Win` 评分更高的问题，同时系统排查并修复同类“专属机制海克斯被泛化规则高估”的问题。
- 直接成功标准：
  - `Karma / Seraphine / Renata` 在 `54 vs 214` 的对比中，`214 Spin To Win` 不再高于 `54 Master of Duality`；
  - `Spin To Win` 这类明确要求“旋转技能”的海克斯，对无旋转技能英雄应落到明显低优先级；
  - 同类高风险海克斯至少覆盖到当前生产池中的专属机制条目，不再只靠通用 `on_skill / damage / poke` 评分；
  - 新增负例回归，后续再出现类似错配会被 `audit:recommendations` 拦下。

## Current State Analysis

### 已确认的直接根因

- `src/data/augments/sixth-batch.ts`
  - `Master of Duality` 目前只有：
    - `triggerTypes: ["on_attack", "on_skill"]`
    - `preferredUsers: ["melee", "attacker", "caster"]`
    - `playstyleFits: ["dps", "burst"]`
  - `Spin To Win` 目前只有：
    - `triggerTypes: ["on_skill"]`
    - `preferredUsers: ["melee", "attacker"]`
    - `playstyleFits: ["dps", "frontline"]`
  - 两者都只有文本描述，没有供规则层直接消费的“专属机制门槛”。

- `src/lib/recommendation/rules.ts`
  - 当前大量加分来自通用逻辑：
    - `on_skill + high_cast_frequency`
    - `rewardTypes.includes("damage")`
    - `trigger_reliability` 中 `on_skill -> skill / high_cast_frequency / haste`
  - 这会让 `Karma / Seraphine / Renata` 这类高频施法辅助，在面对任何“技能伤害型高条件海克斯”时都吃到一层泛化加分。
  - 但规则里没有任何一条显式判断：
    - 英雄是否真的有“旋转技能”；
    - 英雄是否真的同时依赖高频普攻和高频施法；
    - 海克斯的专属收益是否只命中了一半，另一半其实几乎空转。

- `src/data/champions/eleventh-batch.ts`
  - `Karma / Seraphine / Renata` 都是：
    - `damagePatterns: ["skill"]`
    - `combatTempos: ["poke"]`
    - `scalingTags: ["ap", "haste", "heal_shield"]`
    - `triggerTraits` 含 `high_cast_frequency`
  - 这解释了为什么她们会被当前通用 `on_skill` 逻辑误抬。
  - 同时，这 3 个英雄都没有任何“旋转技能”相关 mechanic。

### 复盘结论

- 这次错排不是查询页排序问题。
  - `src/lib/recommendation/catalog.ts` 只是按 `totalScore` 排序，没有额外偏置。
  - `src/components/query-panel.tsx` 只是展示排序结果，没有重算分数。
- 真正问题在评分模型：
  - `Spin To Win` 的“必须有旋转技能”没有进入机器可判定规则；
  - `Master of Duality` 的“双向成长必须都能兑现”也没有进入机器可判定规则；
  - 最后变成“谁更像一个泛用 on_skill damage augment，谁就更高分”。

### 已发现的同类风险点

- `src/data/augments/second-batch.ts`
  - `Bread And Butter` 是明显的 `Q` 专属海克斯，但当前规则只把它当成一般 `on_skill` 海克斯看待。
- `src/data/augments/first-batch.ts`
  - `ADAPt` 是“额外攻击力 -> 法强”的转化海克斯，当前缺少“是否既有 AD 来源、又能稳定吃 AP”的双向校验。
- `src/data/augments/fourth-batch.ts`
  - `escAPADe` 是“法强 -> 攻击力”的反向转化海克斯，当前也缺少“是否真有 AD 兑现路径”的校验。
- `src/data/augments/first-batch.ts`
  - `Skilled Sniper` 已有相对明确的专属规则，是当前做得更对的一类；可以作为这次修复的参照模板。

### 已发现的回归闸门缺口

- `scripts/audit-recommendations.mjs`
  - 目前有正例，如 `Master Yi -> Master of Duality`；
  - 但没有任何负例去防守：
    - “辅助英雄不该被 `Spin To Win` 高估”
    - “非攻技循环英雄不该被 `Master of Duality` 高估”
    - “非 Q 核心英雄不该被 `Bread And Butter` 高估”
- 另外，这个文件里还残留了旧的 `Nami` 基准样例，仍引用了已被阻断的 `id: "5"`，说明回归资产本身也需要顺手收口。

## Proposed Changes

### 1. 给海克斯画像补机器可判定的“专属约束元数据”

- 修改文件：`src/lib/recommendation/types.ts`
- 修改文件：`src/data/augments/first-batch.ts`
- 修改文件：`src/data/augments/second-batch.ts`
- 修改文件：`src/data/augments/fourth-batch.ts`
- 修改文件：`src/data/augments/sixth-batch.ts`
- 做法：
  - 为 `AugmentProfile` 增加可选的规则提示字段，建议采用轻量结构，例如：
    - `ruleHints.requiredAbilityMechanicsAny?: string[]`
    - `ruleHints.requiredRangeType?: "melee" | "ranged"`
    - `ruleHints.requiresAttackSkillLoop?: boolean`
    - `ruleHints.requiresAttackPattern?: boolean`
    - `ruleHints.requiresApScaling?: boolean`
    - `ruleHints.requiresAdScaling?: boolean`
    - `ruleHints.requiresAllyBuffOrShield?: boolean`
  - 先只给当前高风险专属海克斯补齐：
    - `Spin To Win`
    - `Master of Duality`
    - `Bread And Butter`
    - `ADAPt`
    - `escAPADe`
- 原因：
  - 把“专属门槛”写在数据层，比继续把条件硬编码进 `rules.ts` 更可维护；
  - 后续新增类似海克斯时，可以直接在画像处声明范围，不再依赖临时补丁。

### 2. 给英雄画像补必要的 ability mechanics 标签

- 修改文件：
  - `src/data/champions/first-batch.ts`
  - `src/data/champions/sixth-batch.ts`
  - 视扫描结果可能补到其他当前生产英雄文件
- 做法：
  - 给确实存在“旋转技能 / 攻技循环 / Q 核心”特征的现有正例英雄补少量 mechanics 标签，避免规则只能靠模糊猜测：
    - 例如 `Darius Q` 增加 `spin`
    - `MasterYi` 补更明确的“普攻与技能循环”标签
    - 若现有生产英雄里存在明确 `Q` 核心英雄，则在对应 `Q` 上补可识别标签
- 原因：
  - 当前 `ChampionAbilityDescriptor.mechanics` 已是开放字符串数组，具备承载能力；
  - 只需给当前已放行的高置信正例补标签，不必重构整库。

### 3. 收紧规则：把“专属机制门槛”放到 baseline / reward / risk 三层

- 修改文件：`src/lib/recommendation/rules.ts`
- 做法：
  - 新增专属判定 helper，按当前生产池优先覆盖以下几类：
    - `isSpinChampion(champion)`：至少要求近战或前排，并有 `spin` 类 ability mechanic
    - `isAttackSkillLoopChampion(champion)`：既有明确普攻兑现路径，也有高频技能循环
    - `canUseAdToApConversion(champion)`：既有 AD 来源/混合来源，也有 AP 收益路径
    - `canUseApToAdConversion(champion)`：既有 AP 来源，也有 AD 兑现路径
    - `isQCoreChampion(champion)`：Q 确实是主要输出或功能循环支点
  - 对应调整分数结构：
    - 专属门槛满足时，给这类海克斯额外正向加权；
    - 专属门槛不满足时，追加明确负向惩罚，而不是只靠 `high_condition` 扣 10 分；
    - 对“只吃到一半收益”的情况，增加中等负向惩罚，而不是当成完全泛用伤害海克斯。
- 本次重点修复的预期：
  - `Karma / Seraphine / Renata` 对 `Spin To Win` 必须吃到明显的专属门槛失败惩罚；
  - 她们对 `Master of Duality` 仍应保留“有 AP 收益”的部分，但因为缺少稳定普攻循环，不应被高估为强解；
  - 在 `54 vs 214` 的纯对比中，`54` 至少应高于 `214`。

### 4. 系统排查当前 20 个生产海克斯里的同类高风险条目

- 修改文件：`src/lib/recommendation/rules.ts`
- 只读复盘已确认优先排查对象：
  - `Spin To Win`
  - `Master of Duality`
  - `Bread And Butter`
  - `ADAPt`
  - `escAPADe`
- 做法：
  - 对当前生产海克斯按两类分层处理：
    - 泛用海克斯：保留现有通用规则
    - 专属机制海克斯：必须有专属门槛
  - 本轮不扩展到所有 augment 都做专属判定，只处理文本机制非常明确、且已经出现误判风险的条目。

### 5. 重建推荐回归：补负例而不是只补正例

- 修改文件：`scripts/audit-recommendations.mjs`
- 修改文件：`docs/recommendation-audit.md`
- 做法：
  - 为本次问题增加直接负例：
    - `Karma`: `54 / 214 / 141`，要求 `214` 不得高于 `54`
    - `Seraphine`: `54 / 214 / 141`，要求 `214` 不得高于 `54`
    - `Renata`: `54 / 214 / 141`，要求 `214` 不得高于 `54`
  - 增加正例防守：
    - 当前生产池内，至少保留一个真正适合 `Spin To Win` 的英雄基准
    - 保留 `Master Yi -> Master of Duality` 的既有正例，避免修坏正向 archetype
  - 增加同类负例：
    - 至少一组“非 Q 核心英雄不应把 `Bread And Butter` 排第一”
    - 至少一组“纯法辅不应把 `ADAPt / escAPADe` 这类转化海克斯排第一”
  - 同步修掉当前脚本里的陈旧样例：
    - `Nami` 基准仍引用已阻断的 `id: "5"`，需改回 allowlist 内合法海克斯组合

### 6. 让查询结果更容易暴露“专属机制没满足”的原因

- 修改文件：`src/lib/recommendation/rules.ts`
- 可能修改文件：`src/components/query-panel.tsx`
- 做法：
  - 在 `RuleReason` 里加入更直白的风险原因，例如：
    - “英雄没有旋转技能，这条海克斯核心收益几乎空转”
    - “英雄缺少稳定普攻循环，只能吃到一半双修成长”
  - 查询页无需重做 UI，但应能把这种负向主因暴露到现有“主要风险”区域。
- 原因：
  - 用户已经能通过点选候选看到问题，说明前端需要继续承担“发现错配”的职责；
  - 修完后要让错配原因能被直接看懂，不只是分数变化。

## Assumptions & Decisions

- 假设 1：
  - 用户的判断口径成立：`Karma / Seraphine / Renata` 明显不应让 `Spin To Win` 在与 `Master of Duality` 的对比中更高。
- 决策 1：
  - 本次不把 `Master of Duality` 简化成“只要有 AP 就适合”，而是把它修成“至少比 `Spin To Win` 更合理，但仍受攻技循环程度约束”。
- 决策 2：
  - 修复重点放在“专属机制海克斯必须有专属门槛”，不重写整套评分框架。
- 决策 3：
  - 先覆盖当前生产池里已确认高风险的 5 类专属海克斯，不在这轮顺手扩展全部 augment 规则。
- 决策 4：
  - 回归必须增加负例；否则以后只测正例，类似错配还会继续漏出去。

## Verification Steps

- 规则验证
  - 对 `Karma / Seraphine / Renata` 分别比较 `54` 与 `214`，确认 `214` 不再高于 `54`。
  - 检查评分 breakdown，确认存在明确的“专属机制不满足”负向原因。
- 回归验证
  - 运行 `npm run audit:recommendations`
  - 确认新增负例通过，且 `Master Yi -> Master of Duality`、`Lux -> Skilled Sniper` 等既有正例不回退。
- 数据验证
  - 确认本轮新增的 augment `ruleHints` 与 champion ability mechanics 只落在已识别的高风险条目上，没有无依据泛化。
- 工程验证
  - 运行 `npm run audit:data`
  - 运行 `npm run lint`
  - 运行 `npm run build`
- 页面验证
  - 在查询页手动点选：
    - `Karma / Seraphine / Renata` + `物法皆修 / 旋转至胜`
  - 确认 `旋转至胜` 不再高于 `物法皆修`，且风险原因可读。
