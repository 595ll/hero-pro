# 查询面板重置与无默认选择计划

## Summary
- 去掉首页查询面板中“默认选中第一个英雄”的行为，首次进入时英雄应为空状态。
- 在查询面板中新增两个独立按钮：`重置英雄` 和 `重置海克斯符文`。
- 按钮各自放在对应选择区域内；`重置英雄` 只清空英雄选择，不清空已选海克斯，确保不影响现有核心推荐流程。

## Current State Analysis
- 当前查询逻辑集中在 `src/components/query-panel.tsx`。
- 英雄初始值由 `useState(championProfiles[0]?.id ?? "")` 决定，因此页面加载后会默认选中第一个英雄。
- 海克斯选择通过 `selectedAugmentIds` 数组维护，初始值已是空数组，不存在默认勾选问题。
- 推荐结果已有保护逻辑：当 `selectedChampionId` 为空或 `selectedAugmentIds.length === 0` 时，`recommendations` 返回空数组，因此移除默认英雄后不会破坏推荐计算。
- 英雄选择区和海克斯选择区目前只有标题、输入控件和内容列表，没有操作按钮；相关样式都在 `src/app/globals.css` 中维护。

## Proposed Changes
### `src/components/query-panel.tsx`
- **What**
  - 将英雄的默认选中状态改为空字符串。
  - 为英雄选择下拉增加一个占位选项，提示用户先手动选择英雄。
  - 新增两个按钮：`重置英雄`、`重置海克斯符文`。
- **Why**
  - 用户明确要求首次进入时不要默认选中英雄。
  - 用户需要快速回到“未选英雄”或“未选海克斯”的状态，而不必手动逐个撤销。
- **How**
  - 将 `selectedChampionId` 的初始值改为 `""`，不再引用 `championProfiles[0]`。
  - 在 `<select>` 顶部增加一个 `value=""` 的占位 `<option>`，文案采用类似“请选择英雄”。
  - 保留现有 `displayedChampion` / `recommendations` / `selectedResults` 的空值保护逻辑，因为它已经兼容未选英雄状态。
  - 在英雄选择卡内增加 `type="button"` 的 `重置英雄` 按钮，点击后执行 `setSelectedChampionId("")`。
  - 在海克斯选择卡内增加 `type="button"` 的 `重置海克斯符文` 按钮，点击后执行 `setSelectedAugmentIds([])`。
  - `重置英雄` 不清空海克斯，保持用户已勾选的海克斯列表，以便切换英雄重新比较。
  - 结果区无需新增复杂逻辑；当英雄为空时，现有逻辑应自然回退到空结果或提示状态。

### `src/app/globals.css`
- **What**
  - 为查询区域新增按钮布局和按钮样式。
  - 保证新增按钮与现有深色面板风格一致，且不会挤坏当前桌面/移动端布局。
- **Why**
  - 查询区当前没有可复用的按钮行样式，需要补齐结构化布局。
  - 本次变更是交互层增强，必须在视觉上保持一致且不影响可读性。
- **How**
  - 为查询区增加一组轻量操作样式，例如按钮行容器和次级按钮样式。
  - 英雄卡和海克斯卡都采用“标题/标签 + 按钮”的紧邻布局，确保用户能直接理解按钮作用范围。
  - 移动端媒体查询下，允许按钮换行或纵向排列，避免窄屏溢出。

## Assumptions & Decisions
- 首次进入页面时，英雄下拉默认显示占位项，而不是自动选择第一个英雄。
- 按钮采用两个独立操作，而不是单个“全部重置”按钮。
- `重置英雄` 放在英雄选择区域内，`重置海克斯符文` 放在海克斯选择区域内。
- `重置英雄` 只清空当前英雄，不改动已勾选海克斯；`重置海克斯符文` 只清空海克斯，不改动当前英雄。
- 不新增确认弹窗、不新增本地持久化、不改动推荐规则和数据结构。

- 代码检查
  - 确认 `src/components/query-panel.tsx` 中 `selectedChampionId` 初始值为 `""`。
  - 确认英雄下拉存在占位选项，且只有用户主动选择后才展示英雄详情。
  - 确认存在两个独立按钮，并分别调用清空英雄与清空海克斯的状态更新逻辑。

  - 选择英雄后，英雄详情正常显示。
  - 点击 `重置英雄` 后，英雄回到未选择状态，同时已勾选海克斯仍然保留。
  - 点击 `重置海克斯符文` 后，海克斯全部清空，但当前英雄保持不变。

  - 如执行页面实测，重点确认移动端布局下按钮不会挤压下拉框或海克斯列表。
