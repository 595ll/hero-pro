# 英雄与海克斯图标接入计划

## Summary

- 目标：为查询区中的英雄与海克斯符文补上图标，同时严格限制影响范围，不改变现有推荐逻辑、验证逻辑和生产数据模型。
- 总体策略：图标只作为展示层能力接入，资源统一镜像到本地 `public/icons/`，页面只读本地静态资源，不在运行时依赖 Riot 或第三方站点。
- 关键边界：
  - 不改 `src/lib/recommendation/*` 的评分、校验、目录结构和消费方式。
  - 不因图标接入而扩写或重构 `ChampionProfile` / `AugmentProfile` 类型。
  - 尽量不批量改动现有批次数据文件；若海克斯现有 `iconPath` 与真实资源扩展名不一致，则通过展示层覆盖表兜底，不改画像结构。

## Current State Analysis

### 已确认的现状

- 查询页唯一英雄/海克斯交互入口是 `src/components/query-panel.tsx`。
- 样式集中在 `src/app/globals.css`，当前没有任何图片组件或图标布局。
- 英雄与海克斯资料入口分别是：
  - `src/data/champions/index.ts`
  - `src/data/augments/index.ts`
- 推荐链路由 `src/lib/recommendation/catalog.ts` 和 `src/lib/recommendation/validation.ts` 消费 `championProfiles` / `augmentProfiles`，当前并不依赖图标渲染。

### 已确认的数据形态

- `src/lib/recommendation/types.ts` 中 `ChampionProfile` 与 `AugmentProfile` 已预留可选字段 `iconPath?: string`。
- 当前所有已接入英雄和海克斯批次数据里都已经写入了 `iconPath`：
  - 英雄形如 `/icons/champions/Lux.png`
  - 海克斯形如 `/icons/augments/205.png`
- 但仓库目前不存在 `public/icons/` 目录，因此这些路径现在是“已写入数据但前台未使用且文件缺失”的状态。

### 已确认的外部来源

- 英雄图标：
  - Riot 官方 `Data Dragon` 文档明确提供 Champion Square Assets，可作为权威静态图标来源。
  - 现有仓库同步脚本 `scripts/sync-source-data.mjs` 已经在使用 `Data Dragon`，因此来源口径与项目现有原则一致。
- 海克斯图标：
  - `arammayhem.com/zh-cn/augments/` 页面可打开，且页面中的海克斯条目 `img src` 可解析出稳定文件名。
  - 该站点不能作为机制事实来源，但可作为图标与中文名完整性校对的辅助来源，符合仓库 `docs/data-source-spec.md` 中对 B 级来源的定位。

### 本次关键风险

- 当前英雄选择控件是原生 `<select>`，无法在选项中稳定展示图标。
- 海克斯源站图片文件名是英文名风格且为 `.webp`，与现有生产数据中的 `/icons/augments/<id>.png` 不一致。
- 如果直接批量改写现有数据文件中的 `iconPath`，会扩大生产数据改动面，违背“限制影响范围”的要求。

## Assumptions & Decisions

- 用户已确认：
  - 英雄图标不仅要显示在详情卡，也要显示在选择列表中。
  - 图标采用本地静态镜像，而不是运行时外链。
- 本次实现决策：
  - 英雄选择从原生 `<select>` 改为“同面板内的自定义按钮列表”，保留当前 `selectedChampionId` 状态与“重置英雄”行为，不改推荐输入输出。
  - 英雄图标直接复用现有 `ChampionProfile.iconPath` 语义，并在 `public/icons/champions/` 补齐本地文件。
  - 海克斯图标不批量改动现有批次数据；增加一个仅供前端展示使用的图标覆盖清单，把海克斯 `id` 映射到本地实际镜像路径。
  - 图标渲染采用 Next/React 的纯展示层方式；若资源缺失，前端回退到文字布局，不阻断现有功能。

## Proposed Changes

### 1. `src/components/query-panel.tsx`

- 目标：把图标接入限定在查询页展示层，不触碰推荐算法和数据组织。
- 具体改动：
  - 将“1. 选择英雄”区域从原生 `<select>` 改为可点击的英雄按钮列表。
  - 英雄列表项展示：
    - 英雄图标
    - 中文名
    - 可选副标题或状态样式
  - 已选英雄详情卡补上英雄头像。
  - 海克斯候选卡补上海克斯图标。
  - “当前优先级最高”卡片补上海克斯图标。
  - 结果列表 `result-item` 顶部补上海克斯图标。
- 实现要求：
  - 继续使用当前的 `selectedChampionId` / `selectedAugmentIds`。
  - `recommendForAugmentCandidates()`、`canScoreInProduction()` 的调用保持原样。
  - 图标只参与展示，不进入排序、过滤或状态判断。

### 2. `src/app/globals.css`

- 目标：为新增图标与自定义英雄列表提供样式，同时不破坏现有布局节奏。
- 具体改动：
  - 新增英雄列表容器、英雄按钮、选中态样式。
  - 新增通用图标尺寸类，至少覆盖：
    - 英雄列表小图标
    - 已选英雄详情卡头像
    - 海克斯候选卡图标
    - 推荐结果卡图标
  - 为图标 + 文本组合增加横向布局类，避免影响现有标签和文案块。
  - 保持移动端断点兼容，避免图标接入后造成候选卡溢出或结果卡换行失控。

### 3. `src/data/icon-overrides.ts` 或等价展示层文件

- 目标：只在展示层解决海克斯本地路径与现有数据 `iconPath` 不一致的问题。
- 具体改动：
  - 新增一个展示层图标覆盖表，提供：
    - `getChampionIconPath(championId, fallbackIconPath?)`
    - `getAugmentIconPath(augmentId, fallbackIconPath?)`
    - 或者等价的 `Record` 映射 + 小型 helper
  - 英雄：
    - 默认返回现有 `profile.iconPath`
    - 仅作为资源缺失时的集中回退点
  - 海克斯：
    - 优先返回本次镜像后的真实本地文件路径
    - 不要求去修改各批次 `iconPath`
- 原因：
  - 沿用 `display-overrides.ts` 的“展示层覆盖”思路
  - 将风险隔离在前台，不把一次资源格式问题扩散到生产画像数据

### 4. `public/icons/champions/*`

- 目标：落地 Riot 官方英雄方形图标本地镜像。
- 具体改动：
  - 按现有数据中的文件命名补齐资源，例如：
    - `public/icons/champions/Lux.png`
    - `public/icons/champions/Ezreal.png`
    - 其余已接入英雄同理
- 来源与口径：
  - 统一来自 Riot `Data Dragon` Champion Square Assets
  - 文件名与现有 `iconPath` 保持一致，避免改动批次数据文件

### 5. `public/icons/augments/*`

- 目标：落地海克斯图标本地镜像，但不反向污染现有画像数据。
- 具体改动：
  - 将当前 allowlist 中实际在前台展示的海克斯图标镜像到本地目录。
  - 文件命名以“展示层覆盖表”实际使用的路径为准，可使用英文名或 `id`，但必须与覆盖表一一对应。
  - 只补当前生产池会展示的海克斯，不扩大到未放行条目。
- 来源与口径：
  - 来源站为 `arammayhem.com`
  - 只作为图标资源与条目完整性辅助来源，不改动任何机制字段

### 6. 可选：`scripts/sync-icon-assets.mjs` 或一次性采集脚本

- 目标：把本地镜像动作收敛成可重复执行的流程，而不是手工零散下载。
- 推荐做法：
  - 读取当前 `championProfiles` / `augmentProfiles`
  - 只为已接入、已展示条目生成下载清单
  - 英雄从 `Data Dragon` 拉取到 `public/icons/champions/`
  - 海克斯按覆盖表或外部映射拉取到 `public/icons/augments/`
- 边界：
  - 脚本只负责资源采集，不改生产数据文件
  - 若实现成本偏高，可先做一次性静态补齐，再决定是否脚本化

## Implementation Notes

### 英雄列表改造方式

- 不做复杂下拉或搜索框，直接在英雄面板内改为可滚动的按钮列表。
- 这样既满足“列表中显示图标”，也比重写下拉交互更稳，且与当前海克斯候选区的交互形态一致。

### 图标回退策略

- 单个图标丢失时：
  - 对应卡片继续显示文字，不阻断点击和推荐结果
  - 控制台可留轻量错误处理，但不新增影响用户体验的报错提示
- 这样可防止资源个别缺失导致整个查询区不可用。

### 为什么不直接批量改数据

- 当前数据已经被推荐系统消费，下午也出现过“图标接入影响数据模型”的问题。
- 本次应把“图标文件格式和路径整理”限制在展示层与静态资源层，避免再次引入生产数据面的回归风险。

## Verification Steps

- 静态检查：
  - 运行 `npm run lint`
  - 如新增 helper 或组件引用，确认不存在 TS / ESLint 报错
- 构建验证：
  - 运行 `npm run build`
  - 确认 `next export` 路径下图片静态资源可被打包引用
- 页面回归：
  - 查询区加载后，英雄列表可正常选择、重置
  - 已选英雄详情卡出现英雄图标
  - 海克斯候选列表、顶部推荐卡、结果列表出现对应海克斯图标
  - 不选择任何英雄/海克斯时，原有空状态与说明文案保持不变
  - 勾选/取消勾选海克斯后，原有推荐排序与说明内容保持不变
- 风险兜底：
  - 抽查至少 3 个英雄与 5 个海克斯图标路径是否命中本地资源
  - 若最终不得不改动任一生产数据文件，则额外执行 `npm run audit:data` 与 `npm run audit:recommendations`

## Out Of Scope

- 不新增新的英雄/海克斯条目
- 不调整推荐规则、评分逻辑和 verified 准入规则
- 不重构首页三段式布局
- 不把第三方图标来源升级为机制事实来源
