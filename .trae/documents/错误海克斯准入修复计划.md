# 错误海克斯准入修复计划

## Summary

- 目标：复盘并修复“非海克斯大乱斗候选符文混入生产海克斯池”的问题，移除当前错误条目，并建立以后不可再次放行的硬闸门。
- 直接问题：`Banner of Command / Combo Master / Accelerating Sorcery / Chauffeur` 已被写入生产海克斯数据与推荐回归，用户确认这些并不属于海克斯大乱斗当前应收录的候选符文。
- 成功标准：
  - 以上错误条目从生产海克斯索引中移除；
  - 推荐回归与文档不再依赖这些条目；
  - 事实层导入脚本不再把“客户端 arena 数据里出现的所有 augment”直接视为可收录海克斯；
  - 数据审计新增“是否属于允许的 Arena 候选池”校验，后续再出现类似条目会直接阻断。

## Current State Analysis

### 已定位的根因

- `scripts/create-intake-batch.mjs`
  - 当前 `resolveAugmentIdentifiers()` 在 `--all-augments` 下直接返回 `datasets.arenaEn.augments` 的全部 `id`。
  - `findAugmentFact()` 只校验“能否在 arena cache 中找到条目”，不校验“该条目是否属于项目允许收录的海克斯候选池”。
- `scripts/sync-source-data.mjs`
  - 当前会完整缓存 `cdragon/arena/en_us.json` 与 `zh_cn.json`，但没有同步任何“可收录白名单”或“排除名单”元信息。
- `scripts/audit-data.mjs`
  - 当前只审计证据链、描述完整度、触发/收益解释性，不审计“条目是否属于 Arena 正式候选池”。
- `src/data/augments/fourth-batch.ts`
  - 含 `Chauffeur`（`专属司机`）。
- `src/data/augments/fifth-batch.ts`
  - 含 `Accelerating Sorcery`（`加速巫术`）。
- `src/data/augments/sixth-batch.ts`
  - 含 `Banner of Command`（`号令之旗`）、`Combo Master`（`连招大师`）。
- `docs/recommendation-audit.md`
  - 当前回归文档仍把上述错误条目当成合法对比候选。
- `scripts/audit-recommendations.mjs`
  - 当前稳定性样例使用了 `Accelerating Sorcery / Banner of Command / Combo Master`，因此即使后续要清退错误海克斯，也必须同步重建基准样例。

### 当前机制缺口

- 项目目前把“有客户端导出文本 + 有模式级证据 + 有镜像条目”错误等同于“允许进入生产海克斯池”。
- 缺少一个独立字段或清单来表达：
  - 该条目是否属于项目当前支持的海克斯候选集合；
  - 若不属于，原因是什么；
  - 审计应如何阻断。
- 这意味着现有 `verified` 只代表“来源可追溯”，不代表“产品范围正确”。

## Proposed Changes

### 1. 建立海克斯准入清单

- 新增文件：`src/data/augments/allowlist.ts`
- 内容：
  - 定义生产允许接入的 augment `id` 集合；
  - 为每个 id 记录基础准入元信息，例如：
    - `id`
    - `name`
    - `sourceOfTruth`
    - `allowedInProduction: true`
  - 对当前已确认错误的条目单独列出阻断集合，例如：
    - `5` `Banner of Command`
    - `16` `Combo Master`
    - `1` `Accelerating Sorcery`
    - `14` `Chauffeur`
- 原则：
  - 执行阶段先做最小可行版本，以当前生产海克斯 `24` 条为基线生成 allowlist；
  - 后续新增海克斯必须先入 allowlist，再允许接入批次文件与生产索引。

### 2. 收紧生产海克斯索引

- 修改文件：`src/data/augments/index.ts`
- 做法：
  - 保留当前批次组织方式；
  - 但导出的 `augmentProfiles` 需要经过 allowlist 过滤，确保即使批次文件里残留错误条目，也无法进入生产推荐。
- 原因：
  - 这是最快的生产层兜底；
  - 即使历史批次文件未来还没完全拆净，生产索引先不再暴露错误条目。

### 3. 清退当前错误条目

- 修改文件：
  - `src/data/augments/fourth-batch.ts`
  - `src/data/augments/fifth-batch.ts`
  - `src/data/augments/sixth-batch.ts`
- 做法：
  - 删除或降级以下条目，确保它们不再被视为当前支持的 Arena 海克斯画像：
    - `Chauffeur`
    - `Accelerating Sorcery`
    - `Banner of Command`
    - `Combo Master`
  - 执行时优先直接从生产批次文件中移除，避免“索引过滤通过但源数据仍误导”的长期技术债。
- 注意：
  - 若这些条目对事实层审查仍有保留价值，执行阶段可转移到 `tmp/` 或单独文档，不再保留在 `src/data/augments/*.ts` 生产目录。

### 4. 给导入脚本补“准入过滤”

- 修改文件：`scripts/create-intake-batch.mjs`
- 做法：
  - 新增对 allowlist / denylist 的读取；
  - `--all-augments` 不再等于 arena cache 全量，而是：
    - 默认仅导出 allowlist 中的条目；
    - 若未来需要事实层全量研究，再显式使用新的参数，例如 `--all-augments-raw`。
  - `findAugmentFact()` 在执行阶段增加校验：
    - 如果目标不在 allowlist 且未显式要求 raw 导出，则直接报错。
- 结果：
  - “全量事实导入”和“生产候选导入”被明确拆开，不再混淆。

### 5. 给源同步/文档补清晰口径

- 修改文件：
  - `scripts/sync-source-data.mjs`
  - `docs/data-source-spec.md`
  - `docs/quality-gate.md`
  - `docs/data-entry-workflow.md`
- 做法：
  - 明确写出：`cdragon/arena/*.json` 是“候选原始池”，不是“生产可收录池”；
  - 生产放行必须额外通过“候选池归属”校验；
  - 明确区分三层：
    - 原始客户端导出池
    - 事实层审查池
    - 生产允许池
- 文档必须新增复盘结论：
  - 此次错误来源于“把原始池当成生产池”；
  - 后续凡是 Arena 增强项进入生产前，必须先经过 allowlist 准入。

### 6. 给数据审计加硬阻断

- 修改文件：`scripts/audit-data.mjs`
- 做法：
  - 审计时加载 `src/data/augments/allowlist.ts`；
  - 对 `src/data/augments/*-batch.ts` 中的每个 `verified` augment 校验：
    - `id` 必须存在于 allowlist；
    - 如不在 allowlist，直接报错并中断。
  - 增加更明确的错误信息：
    - `Augment 5: verified but not in Arena production allowlist`
    - 中文版错误信息可同步输出，便于快速定位。
- 结果：
  - 以后即使人工再误录，也过不了 `audit:data`。

### 7. 重建推荐回归样例

- 修改文件：
  - `scripts/audit-recommendations.mjs`
  - `docs/recommendation-audit.md`
- 做法：
  - 移除依赖错误海克斯的样例：
    - `Swain` 不再对比 `Accelerating Sorcery`
    - `Soraka` 不再对比 `Banner of Command`
    - `Brand` 不再对比 `Combo Master`
  - 换成当前确认属于生产海克斯池的合法候选组合。
- 要求：
  - 新样例仍需覆盖原先想表达的 archetype 目标：
    - 法坦生存
    - 附魔辅助
    - 连段法师
  - 但不再使用任何被清退条目。

## Assumptions & Decisions

- 假设 1：
  - 用户指出的 `号令之旗 / 连招大师 / 加速巫术 / 专属司机` 为错误收录，执行阶段应以“从生产支持范围中移除”为准，不再尝试为其辩护或保留。
- 决策 1：
  - 修复重点不是争论单个条目文本来源，而是把“来源可信”和“范围正确”拆成两道门。
- 决策 2：
  - 生产索引过滤 + 批次源数据清退 + 审计硬阻断三层同时做，避免只修一层后别处继续漏。
- 决策 3：
  - `--all-augments` 的语义应改为“生产候选全量”，而不是“arena 原始导出全量”。
- 决策 4：
  - 回归样例必须与 allowlist 同口径，绝不允许审计脚本继续引用已被判定错误的条目。

## Verification Steps

- 数据层验证
  - 确认 `src/data/augments/index.ts` 导出的 `augmentProfiles` 中不再包含 `1 / 5 / 14 / 16`。
  - 检查 `src/data/augments/*-batch.ts` 中已无这些条目，或至少不再处于生产接入路径。
- 脚本验证
  - 运行 `npm run audit:data`，确认若手工塞入 allowlist 之外的 verified augment 会被阻断。
  - 运行一次 `npm run intake:batch -- --all-augments`，确认导出的不再是 arena 原始全量，而是 allowlist 全量。
- 推荐验证
  - 运行 `npm run audit:recommendations`，确认回归样例已全部切换为合法海克斯组合并通过。
- 工程验证
  - 运行 `npm run lint`
  - 运行 `npm run build`
- 复盘验证
  - 文档中明确写出：
    - 原始池不等于生产池
    - allowlist 是海克斯准入硬闸门
    - 以后新增海克斯必须先补 allowlist，再进画像与回归
