# 第七批高置信英雄放行计划

## Summary

- 本轮目标不是继续追求“全量一起上线”，而是在不降低数据底座质量与推荐准确性的前提下，从全量事实层审查包中放行第一批高置信 archetype。
- 放行范围收缩为 `Janna / Milio / Nami` 三个附魔/保人辅助英雄；本轮不放行新的海克斯画像。
- 选择这一策略的原因是：当前全量英雄事实层已完整，且这三名英雄的官方技能文本能直接支撑 `heal_shield / peel / ally_buff / lockdown` 等画像；相对地，部分新海克斯草稿仍存在 `数值%` 这类未还原占位，若直接放行会增加描述不准确与推荐误导风险。
- 本轮执行成功的标准：
  - 新增三名 `verified` 英雄且证据链、技能机制、画像标签全部可解释
  - 不新增任何描述未完全还原的新海克斯到生产索引
  - 支持型英雄在既有 `verified` 海克斯候选中的排序稳定且可解释
  - `audit:data`、`audit:recommendations`、`lint`、`build` 全部通过

## Current State Analysis

- 当前生产英雄索引只接入了 `src/data/champions` 下前六批，合计 `18` 个英雄；生产海克斯索引只接入了前六批，合计 `24` 个海克斯。
- `scripts/audit-data.mjs` 已具备：
  - 批次文件自动发现
  - `verified` 证据完整性校验
  - 英雄画像与技能/生存/功能标签的一致性校验
  - 海克斯触发与风险说明的一致性校验
- `scripts/audit-recommendations.mjs` 已具备 `8` 组推荐回归样例，覆盖支持、普攻、法坦、控制前排、连段法师与混合成长等方向。
- `tmp/intake-batches/full-facts-intake-2026-04-25` 已存在全量事实层审查包，且其中：
  - `Janna / Milio / Nami` 已有完整中文技能事实草稿，可直接作为人工补画像底稿
  - `First-Aid Kit` 等新海克斯草稿仍带有 `数值%` 占位，说明“新海克斯直接放行”当前仍不够稳
- 因此，本轮最稳妥的推进方式是：只放行新英雄，不放行新海克斯；继续使用现有已验证海克斯作为推荐对照集。

## Proposed Changes

### 1. 新增第七批英雄画像

- 文件：`src/data/champions/seventh-batch.ts`
- 做什么：
  - 基于全量事实层审查包，为 `Janna`、`Milio`、`Nami` 手工补全正式 `ChampionProfile`
  - 每个英雄都要补齐：
    - `damagePatterns`
    - `combatTempos`
    - `scalingTags`
    - `triggerTraits`
    - `survivalStyles`
    - `utilityTags`
    - `coreMechanics`
    - `antiPatterns`
    - 5 个技能的 `mechanics`
  - `verification.status` 直接按 `verified` 录入，但前提是两条 `riot-ddragon` 证据齐全且画像可回溯到技能文本
- 为什么：
  - 这三个英雄的角色边界稳定，属于支持型 archetype 中最容易形成一致判断的一类
  - 现有生产海克斯里已经有 `全心为你`、`号令之旗` 等支持向条目，不需要同步新增海克斯也能形成有效推荐
- 怎么做：
  - 以 `src/data/champions/sixth-batch.ts` 中 `Soraka` 的写法为模板
  - 使用 `tmp/intake-batches/full-facts-intake-2026-04-25/champion-drafts.ts` 里的事实草稿作为基础
  - 其中画像边界建议固定为：
    - `Janna`：偏 `peel / disengage / shield / move_speed / lockdown`
    - `Milio`：偏 `heal_shield / ally_buff / range_extension / peel`
    - `Nami`：偏 `heal_shield / poke / ally_buff / lockdown`

### 2. 接入英雄索引，不扩海克斯索引

- 文件：`src/data/champions/index.ts`
- 做什么：
  - 引入 `seventhBatchChampionProfiles`
  - 将其并入 `championProfiles`
- 为什么：
  - 只把本轮确认过的英雄接入生产数据
- 怎么做：
  - 保持现有批次组织方式，新增 import 并加入展开数组

- 文件：`src/data/augments/index.ts`
- 做什么：
  - 本轮不改
- 为什么：
  - 避免把仍存在占位描述的新海克斯带入生产

### 3. 收紧数据审计，对未还原占位做显式阻断

- 文件：`scripts/audit-data.mjs`
- 做什么：
  - 为海克斯 `verified` 条目新增文本质量校验
  - 至少阻断以下情况进入生产：
    - `summary` 或 `officialDescription` 含 `数值`
    - `summary` 或 `officialDescription` 为空白或只剩模板化残句
  - 对 `provisional` 草稿保持宽松，不阻断事实层审查包生成
- 为什么：
  - 当前已经确认全量草稿中存在未还原数值占位，如果不把这条规则写进审计，后续容易误放行
- 怎么做：
  - 在 `auditAugment()` 中，仅对 `verification.status === "verified"` 的海克斯加上述校验

### 4. 细化支持型推荐规则，避免“支持英雄被攻击型海克斯误顶”

- 文件：`src/lib/recommendation/rules.ts`
- 做什么：
  - 补强支持型英雄的显式正负向约束
  - 增加的方向包括：
    - `heal_shield`、`ally_buff`、`peel`、`disengage` 对 `All For You`、`Banner of Command` 这类收益的正向加权
    - 纯支持英雄对 `on_attack`、高单兵输出导向海克斯的额外降权
    - `move_speed / peel / range_extension` 对保人、拉扯型辅助的加权区分
- 为什么：
  - 本轮新增的三个英雄都属于“伤害不是主要兑现点”的 archetype；若不补负向约束，容易出现输出向海克斯被抬高的假阳性
- 怎么做：
  - 延续现有规则文件里的“信号匹配 + 风险扣分”模式
  - 只做支持 archetype 相关的最小增量，不重写整套打分逻辑

### 5. 扩充推荐回归，固定支持 archetype 的稳定性

- 文件：`scripts/audit-recommendations.mjs`
- 做什么：
  - 新增 3 组回归样例：
    - `Janna` 候选为 `5 / 141 / 87`，第一名允许 `5` 或 `141`，禁止 `87`
    - `Milio` 候选为 `141 / 5 / 87`，第一名允许 `141` 或 `5`，禁止 `87`
    - `Nami` 候选为 `141 / 5 / 113`，第一名允许 `141` 或 `5`，禁止 `113`
  - 三组样例的最低结论统一要求至少为 `good_pick`；若规则足够稳定，可以提升到 `strong_pick`
- 为什么：
  - 支持英雄往往存在多个合理第一名，不能用“只允许唯一第一”来硬钉
  - 但必须明确排除明显错误的输出型条目排第一
- 怎么做：
  - 复用当前脚本已支持的 `expectedTopAugmentIds` 与 `disallowedTopAugmentIds` 机制

### 6. 更新文档，明确“第七批先放英雄、延后新海克斯”

- 文件：`docs/quality-gate.md`
- 做什么：
  - 增补一条明确规则：任何带未还原数值占位的海克斯文本不得进入 `verified`
- 文件：`docs/data-entry-workflow.md`
- 做什么：
  - 补充说明：全量事实层导入后，优先放行“画像可直接由技能文本解释”的英雄 archetype；描述未完全还原的海克斯继续留在 `provisional`
- 为什么：
  - 让后续继续扩数时遵循同一质量口径，而不是本轮临时判断

## Assumptions & Decisions

- 决策：本轮生产放行范围仅限英雄，不包含新的海克斯画像。
- 决策：第七批只选择 `Janna / Milio / Nami`，不同时放行更多支持英雄，避免在一轮内扩大人工复核面。
- 决策：支持英雄的推荐回归允许多个等价第一名，但必须显式排除明显错误第一名。
- 决策：新增海克斯放行的前置条件是文本描述已从事实层草稿还原为可读、可解释、无占位的正式文案。
- 假设：既有生产海克斯 `5 / 141 / 87 / 113` 的画像与当前推荐脚本仍然稳定，可作为本轮支持 archetype 的对照集。
- 假设：全量事实层审查包中的 `Janna / Milio / Nami` 技能草稿可直接作为正式录入底稿，不需要额外抓源。

## Verification Steps

- 代码与数据改动后，按顺序执行：
  - `npm run audit:data`
  - `npm run audit:recommendations`
  - `npm run lint`
  - `npm run build`
- 人工抽样复核：
  - 检查 `src/data/champions/seventh-batch.ts` 中三名英雄的每个技能 `mechanics` 是否都能回溯到官方技能文本
  - 检查三名英雄的 `coreMechanics` 与 `antiPatterns` 是否存在“支持型结论但文本无支撑”的问题
  - 在查询页手工抽查：
    - `Janna + [5, 141, 87]`
    - `Milio + [141, 5, 87]`
    - `Nami + [141, 5, 113]`
  - 目标是：
    - 第一名为支持向海克斯
    - 不出现明显输出向海克斯排第一
    - 理由与风险提示能解释排序
