# 数据质量与推荐稳定性加固计划

## Summary

- 本轮不动 UI，范围固定为“数据质量 + 推荐稳定性 + 推荐准确性”。
- 结合当前仓库状态，最优先的不是继续改页面，而是补齐两类质量缺口：
  - 生产验证闸门与数据审计口径不一致
  - 缺少可重复运行的推荐回归校验，无法稳定判断规则调整是否导致推荐漂移
- 执行目标是让“能进正式推荐的数据”与“能通过审计的数据”保持一致，并且为核心英雄/海克斯组合建立一套可回归检查的基准集。

## Proposed Changes

### 1. 对齐生产验证闸门与审计口径

**文件**

- [validation.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/lib/recommendation/validation.ts)
- [audit-data.mjs](file:///c:/Users/29403/Documents/trae_projects/hero-pro/scripts/audit-data.mjs)

**做什么**

- 把生产推荐闸门从“有任一一级来源即可”收紧到和审计规则一致。
- 英雄与海克斯分别使用明确的最低证据组合，而不是只看泛化的 primary source。

**为什么**

- 现在存在“数据审计不过，但理论上仍可能被生产验证放行”的风险。
- 对用户来说，正式推荐必须和数据质量闸门完全一致，否则会出现口径分裂。

**怎么做**

- 在 `validation.ts` 中显式拆分：
  - 英雄：`verified` 且必须具备 `riot-ddragon`
  - 海克斯：`verified` 且必须同时具备 `official-client / riot-support / community-mirror`
- 把当前通用 `PRIMARY_SOURCES` 宽口径逻辑改为更接近审计脚本的结构化校验。
- 同步整理 `audit-data.mjs` 中相关判定逻辑，尽量抽成与生产闸门一致的规则表达，减少未来二次偏移。

### 2. 扩充数据审计为“结构一致性审计”

**文件**

- [audit-data.mjs](file:///c:/Users/29403/Documents/trae_projects/hero-pro/scripts/audit-data.mjs)

**做什么**

- 在现有唯一性和字段完整性基础上，再补一层“画像标签与内容是否自洽”的结构审计。

**为什么**

- 当前审计更偏“有没有填”，还不够判断“填得是否自洽”。
- 推荐准确性不只依赖证据来源，也依赖标签有没有和技能机制、描述文本对上。

**怎么做**

- 为英雄增加最小一致性检查，例如：
  - `heal_shield` 英雄至少要出现 `heal / shield / ally_protect / buff` 一类技能机制或生存/功能标签支撑
  - `attack_speed / crit` 英雄至少要具备 `attack` 或 `enhanced_attack` 方向画像支撑
  - `frontline` 生存风格与 `tank`/近身站场描述之间不能完全脱节
- 为海克斯增加最小一致性检查，例如：
  - `on_attack` 海克斯必须有攻击触发导向的 `preferredUsers / positiveSignals`
  - `utility` 或 `survival` 海克斯必须有能解释其收益兑现或风险边界的信号文案
  - 高条件海克斯必须有对应 `trapSignals`
- 保持规则尽量“硬约束 + 少量高价值一致性检查”，不做脆弱的自然语言全文解析。

### 3. 新增推荐回归校验脚本

**文件**

- `scripts/audit-recommendations.mjs`
- [package.json](file:///c:/Users/29403/Documents/trae_projects/hero-pro/package.json)

**做什么**

- 新增一个专门的推荐回归脚本，用固定的英雄-候选海克斯样例集，断言：
  - 第一名是谁
  - 是否至少达到预期结论级别
  - 某些明显不适配项是否不会排到前面

**为什么**

- 这是当前“推荐稳定性”最缺的一环。
- 规则越改越多后，仅靠人工点页面很难发现排序回归或误伤已有强匹配关系。

**怎么做**

- 在脚本里直接复用：
  - [catalog.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/lib/recommendation/catalog.ts)
  - [champions/index.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/data/champions/index.ts)
  - [augments/index.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/data/augments/index.ts)
- 基准样例优先覆盖已人工验证过、方向最明确的组合：
  - `Lulu` 候选含 `All For You / Sonic Boom / Typhoon`
  - `Vayne` 候选含 `Typhoon / Mystic Punch / All For You`
  - `Swain` 候选含 `Big Brain / Accelerating Sorcery / All For You`
  - `Leona` 候选含 `Slap Around / Goredrink / Skilled Sniper`
  - `Lux` 候选含 `Skilled Sniper / All For You / Goredrink`
- 脚本输出应直观指出：
  - 哪个断言失败
  - 实际排序前几名
  - 便于后续定位是哪条规则引发漂移
- 在 `package.json` 中新增脚本，例如：
  - `audit:recommendations`

### 4. 收紧规则层的稳定性边界

**文件**

- [rules.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/lib/recommendation/rules.ts)

**做什么**

- 在不改返回结构的前提下，补一轮“冲突约束”和“误判降权”。

**为什么**

- 当前规则以加分为主，风险更像“能不能再加一点分”，但对明显不匹配的组合缺少足够的抑制。
- 仅靠正向加分，容易出现“很多组合都被推成推荐”的情况，稳定性会下降。

**怎么做**

- 重点增加有限的负向规则，而不是继续无限叠正向规则：
  - `team_comp_dependent` 海克斯遇到明显单兵作战英雄时加大风险扣分
  - `on_attack` 海克斯遇到纯技能英雄时增加更明确的负向抑制
  - 辅助/保人型海克斯遇到自我作战画像时增加负向抑制
  - 高条件海克斯在英雄没有对应触发稳定性时，除风险扣分外再降低基线分
- 保持阈值体系不变：
  - `strong_pick / good_pick / situational / avoid`
- 优先服务“减少假阳性推荐”，而不是追求所有组合都能打出大分差。

## Assumptions & Decisions

- “先不考虑 UI 问题”理解为：本轮不修改 [page.tsx](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/app/page.tsx)、[query-panel.tsx](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/components/query-panel.tsx)、[globals.css](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/app/globals.css)。
- 本轮优先解决“质量口径一致性”和“推荐稳定性可回归”，而不是继续扩第六批数据。
- 推荐回归脚本采用“基准样例断言”而不是引入完整测试框架，避免在当前仓库里过度工程化。
- 本轮不新增后端、不引入数据库，也不改动现有前端交互结构。

2. 运行新增的推荐回归审计
   - `npm run audit:recommendations`

   - `npm run lint`
4. 运行生产构建
