# 海克斯列表多列紧凑化计划

## Summary

- 目标：修正海克斯列表当前又接近“一行一个/一行很少几个”的展示问题，把海克斯候选区改成更紧凑的多列列表，并确保桌面端尽量达到一行 3 个以上。
- 成功标准：
  - 海克斯列表在桌面端显示为紧凑多列网格，优先达到一行 3 个以上。
  - 卡片只保留图标、中文名、等级，不再展示过多描述。
  - 改动范围严格限制在海克斯列表选择区，不影响英雄区、结果区和推荐逻辑。
- 约束：
  - 不改 `src/lib/recommendation/*`
  - 不改数据模型
  - 不改英雄列表与详情区
  - 不改结果区内容结构

## Current State Analysis

### 已确认现状

- 海克斯列表渲染位于 `src/components/query-panel.tsx` 的海克斯选择区：
  - 容器：`augment-scroll-area`
  - 列表：`augment-list`
  - 单项：`augment-item`
- 当前每个海克斯卡片展示了：
  - 图标
  - 中文名
  - 英文名
  - 摘要描述
  - 等级 tag
- 当前 `src/app/globals.css` 中：
  - `.augment-list` 已经是 `display: grid`
  - 其列定义为 `grid-template-columns: repeat(auto-fill, minmax(220px, 1fr))`
  - `.augment-item` 是较宽的纵向卡片，包含较长文案

### 根因判断

- 海克斯列表“看起来又成了一行一个/密度很低”，核心原因不是 grid 失效，而是：
  - 每列最小宽度被设为 `220px`
  - 卡片里仍保留较长摘要内容
  - 当前海克斯列表位于查询区右侧半宽面板内，可用宽度有限
- 这三点叠加后，桌面端通常只能放下 1-2 列，无法达到“一行 3 个以上”。

## Assumptions & Decisions

- 用户已确认海克斯卡片采用最紧凑方案：
  - 保留 `图标 + 中文名 + 等级`
  - 不保留英文名
  - 不保留摘要描述
- 本次采用最小影响策略：
  - 只改海克斯候选列表卡片内容与海克斯列表样式
  - 不调整英雄列表
  - 不调整结果区卡片
  - 不调整数据来源和选择逻辑
- 布局策略：
  - 桌面端通过更小的最小列宽实现 3 列以上
  - 小屏端继续做合理降级，不强行维持 3 列

## Proposed Changes

### 1. `src/components/query-panel.tsx`

- 仅修改海克斯候选列表 `augment-list` 中的单项内容。
- 当前海克斯按钮内将删除：
  - 英文名 `augment.name`
  - 摘要 `augment.summary`
- 保留：
  - 海克斯图标
  - 中文名 `augment.localizedName ?? augment.name`
  - 等级 tag
- 结构调整方向：
  - 将卡片改成更紧凑的“图标 + 名称”头部
  - 等级 tag 继续保留，但采用更省空间的布局
- 保持不变：
  - `selectedAugmentIds`
  - `toggleAugment()`
  - 搜索与等级筛选
  - 空状态逻辑

### 2. `src/app/globals.css`

- 只调整海克斯列表相关样式：
  - `.augment-list`
  - `.augment-item`
  - 海克斯卡片内部与其直接相关的排版类
- 目标样式变化：
  - 将 `.augment-list` 改成更高密度网格，列宽下探到能在当前右侧面板稳定容纳 3 列以上
  - 将 `.augment-item` 改成更紧凑的小卡片
  - 减少内边距、间距和文本占位
  - 让图标与中文名成为主要视觉信息
- 明确边界：
  - 不改 `.result-list`
  - 不改 `.top-pick-card`
  - 不改英雄区相关类

## Interaction Impact

- 用户仍然通过点击海克斯卡片进行勾选/取消勾选。
- 搜索和等级筛选仍然只影响候选区可见项。
- 因为卡片更紧凑，用户能在同一屏看到更多海克斯，选择效率更高。
- 结果区仍承担“解释为什么推荐”的职责，因此候选区不再重复展示摘要，符合当前“影响范围只在海克斯列表”的要求。<mccoremem id="01KQ1NSSRHM75XT49G7QJMDH88|03g0d7kwgycl1cm3c3qo4n8zd" />
