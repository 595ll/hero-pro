# 卡莎错误推荐复盘计划

## Summary

- 目标：复盘并修正 `Kai'Sa` 在以下两组候选中的错误推荐：
  - `Executioner` vs `All For You`
  - `Executioner` vs `Slap Around`
- 成功标准：
  - 能明确解释当前错误推荐为何发生；
  - 修正范围只限本次问题直接相关的数据/规则/回归；
  - 修正后 `Kai'Sa` 不再把 `All For You`、`Slap Around` 排在 `Executioner` 之前；
  - 不影响现有其他已通过的基准样例；
  - 必须补回归并通过 `audit:data`、`audit:recommendations`、`lint`、`build`。

## Current State Analysis

- 当前 `Kai'Sa` 画像位于 `src/data/champions/twelfth-batch.ts`。
- 她的关键标签是：
  - `damagePatterns = hybrid`
  - `combatTempos = dps + burst`
  - `scalingTags = ad + ap + attack_speed`
  - `triggerTraits = enhanced_attack + multi_hit + all_in`
  - `survivalStyles = dash + shield + stealth`
  - `utilityTags = mobility`
- 她没有：
  - `heal_shield`
  - `peel`
  - `engage`
  - `lockdown`
  - 任何 `stun / root / knockup / pull / ground` 一类控制技能机制
  - 任何 `ally_protect / buff / heal` 一类对友军施加治疗护盾的能力

- 当前三条海克斯画像位置：
  - `Executioner` 在 `src/data/augments/seventh-batch.ts`
  - `All For You` 在 `src/data/augments/fifth-batch.ts`
  - `Slap Around` 在 `src/data/augments/third-batch.ts`

- 只读复盘已确认当前实测结果：
  - `Executioner`：`78 / good_pick`
  - `All For You`：`81 / strong_pick`
  - `Slap Around`：`82 / strong_pick`

- 当前错误的根因不是单点，而是 3 个机制叠加：

### 1. `All For You` 被“自保能力”误判成“团队保护能力”

- 在 `src/lib/recommendation/rules.ts` 中，`reward_conversion` 存在过宽逻辑：
  - 只要海克斯含 `utility`，且英雄存在任一 `utilityTags`，或 `survivalStyles` 含 `shield/heal`，就直接给 `+8` 的“功能收益可兑现”；
  - 这会让 `Kai'Sa` 因为有 `mobility` 和自保 `shield`，吃到本应偏向保人/辅助的 `utility` 分。
- 同文件中，`survival` 也有过宽逻辑：
  - 只要英雄有任意 `survivalStyles`，就直接给 `+8` 的“容错收益可兑现”；
  - `All For You` 是“对友军施加治疗和护盾更强”，但现在系统把 `Kai'Sa` 的自保型护盾也当成了可以兑现这条海克斯的理由。
- 结果：
  - `Kai'Sa` 会因为“有盾、有位移、有 AP”被系统当成能吃 `All For You` 的半个功能型英雄；
  - 这与海克斯原文“对友军施加治疗和护盾”不一致。

### 2. `Slap Around` 缺少专属门槛，导致无控制英雄也能吃泛伤害分

- `Slap Around` 当前画像只有：
  - `triggerTypes = on_cc`
  - `rewardTypes = damage`
  - 但没有 `ruleHints.requiredAbilityMechanicsAny`
- `rules.ts` 对 `on_cc` 的触发稳定性只会在英雄有 `cc_chain / lockdown` 时加分，不会在没有控制时主动扣分。
- 同时，`Kai'Sa` 仍会吃到大量泛伤害基线：
  - 节奏匹配
  - 使用者画像匹配
  - 法术画像贴合
  - 伤害收益可兑现
  - 爆发玩法加成
- 结果：
  - 即使 `Kai'Sa` 没有任何定身/缚地/硬控，她对 `Slap Around` 也不会因为“核心触发完全不存在”而被重罚；
  - 最终只靠泛伤害逻辑就把 `Slap Around` 顶到了 `82`。

### 3. `Executioner` 的专属门槛过窄，错误压低了 `Kai'Sa`

- `Executioner` 当前在 `src/data/augments/seventh-batch.ts` 里配置了：
  - `ruleHints.requiredAbilityMechanicsAny = ["reset", "execute", "takedown_reward", "combo_reset"]`
- 这会让没有这些显式机制的英雄直接吃到 `-18` 的“专属技能机制缺失”。
- 但 `Executioner` 的文本是双收益：
  - 对低血目标增伤
  - 参与击杀后刷新基础技能
- `Kai'Sa` 虽然不是经典 `resetter`，但她确实能稳定利用“低血增伤”这半条核心收益，且自身也有：
  - `burst`
  - `all_in`
  - 收残局能力
- 当前规则把它近似当成“必须是带重置/斩杀文本的英雄才配拿”，门槛明显过窄，因此把 `Executioner` 压到了 `78`。

## Proposed Changes

### 1. 收紧 `All For You` 的兑现条件，只承认“对队友的保护与增益”

- 修改 `src/lib/recommendation/rules.ts`
- 目标：
  - 让 `All For You` 这类海克斯只对真正能给队友施加治疗/护盾/增益的英雄加分；
  - 不再让单纯“自己有盾、自己有位移”的英雄吃到 `utility/survival` 奖励。
- 具体方向：
  - 收紧 `utility` 的通用加分：
    - 不能再仅因为 `utilityTags.length > 0`、`survivalStyles.includes("shield")` 就给功能收益分；
    - 至少要命中 `ally_protect / buff / heal` 或明确的 `heal_shield + peel/disengage` 组合。
  - 收紧 `team_comp_dependent` 的容忍条件：
    - 当前 `riskPenalty` 里把任意 `shield` 都视为联动支点，这会误放过 `Kai'Sa`；
    - 应区分“自保护盾”和“对友军保护”。
  - 如有必要，为 `All For You` 增加更明确的 `ruleHints`，把“对友军施加治疗/护盾”写成硬门槛，而不是仅依赖泛规则。

### 2. 给 `Slap Around` 增加控制专属门槛

- 优先修改 `src/data/augments/third-batch.ts`
- 最小方案：
  - 给 `Slap Around` 增加 `ruleHints.requiredAbilityMechanicsAny`
  - 门槛应围绕真实触发文本设置，如：
    - `stun`
    - `root`
    - `knockup`
    - `taunt`
    - `pull`
    - `ground`
- 这样会让 `Kai'Sa` 这类完全没有控制技能机制的英雄，直接吃到明确 mismatch 惩罚。
- 默认不优先改全局 `on_cc` 规则；
  - 先用最小专属门槛修复这条海克斯，避免波及无关条目。

### 3. 放宽 `Executioner` 的门槛，不再把它等同于“纯重置专属”

- 修改 `src/data/augments/seventh-batch.ts`
- 目标：
  - 保留它对收割/残局英雄的偏好；
  - 但不再要求英雄必须显式拥有 `reset / execute / combo_reset` 文本才算能用。
- 具体方向：
  - 删除当前过窄的 `requiredAbilityMechanicsAny` 硬门槛，或改成更宽的门槛；
  - 让 `Executioner` 主要依赖：
    - `triggerTypes = on_kill + on_low_hp`
    - `preferredUsers = attacker / caster / resetter`
    - 英雄的 `burst / all_in / attacker / hybrid` 等泛画像
  - 如果仍需保留专属约束，应改成“偏好收割/进场/爆发”，而不是“必须自带技能重置文本”。

### 4. 补回归，锁住这次问题

- 修改 `scripts/audit-recommendations.mjs`
- 至少新增 3 组样例：
  - `Kai'Sa executioner vs all-for-you`
    - 目标：`32` 必须高于 `141`
  - `Kai'Sa executioner vs slap-around`
    - 目标：`32` 必须高于 `136`
  - `Kai'Sa all-for-you mismatch`
    - 目标：`141` 不能排第一，或最低只到 `situational/avoid`
- 如果修正 `Slap Around` 专属门槛后，`Kai'Sa` 对它能直接掉到 `avoid`，应把这个断言写进回归。
- 同时必须重新跑现有样例，特别是：
  - `Lulu / All For You`
  - `Leona / Slap Around`
  - `Darius / Executioner`
  以确保本次修复不伤到旧正确样本。

## Assumptions & Decisions

- 决策：这次优先修复“规则误判”与“海克斯门槛设计不当”，不改 `Kai'Sa` 英雄画像本身，因为当前画像与技能事实基本一致。
- 决策：优先采用“最小改动”：
  - `All For You` 误判主要改规则；
  - `Slap Around` 误判优先改该条海克斯的专属门槛；
  - `Executioner` 误判优先改该条海克斯的门槛，不先大改全局。
- 决策：不扩大到其它英雄或其它海克斯的重做，只围绕本次 `Kai'Sa` 误判闭环修复。
- 假设：当前工作区已有未提交修改，执行时必须避免覆盖无关文件中的现有变更。

## Verification Steps

1. 复现并确认修复前问题：
  - `Kai'Sa` 对 `32/141`
  - `Kai'Sa` 对 `32/136`
2. 完成最小修复后，运行：
  - `npm run audit:data`

  - `Kai'Sa` 不再把 `Slap Around` 排在 `Executioner` 之前
  - `Lulu` 仍优先 `All For You`
  - `Leona` 仍优先 `Slap Around`
  - `Darius` 仍能合理命中 `Executioner`
