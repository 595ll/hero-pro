# 三条海克斯最小补充计划

## Summary

- 本次只补 3 条生产海克斯符文：`Executioner`、`Dashing`、`Courage of the Colossus`。
- 目标是在不做多余修改的前提下，把这 3 条条目以 `verified` 口径接入生产池，并补齐必要回归。
- 成功标准：
  - 三条海克斯满足生产证据要求与数据审计要求；
  - 只改与这 3 条条目直接相关的最小文件集合；
  - 必须补推荐回归并通过 `audit:data`、`audit:recommendations`、`lint`、`build`；
  - 不改无关英雄画像、不扩大其它海克斯范围、不顺手做额外规则重构。

## Current State Analysis

- 当前生产海克斯入口由 `src/data/augments/index.ts` 聚合，并通过 `src/data/augments/allowlist.ts` 过滤；因此任何新海克斯进入生产都至少需要：
  - 新增画像数据；
  - 进入 allowlist；
  - 通过数据审计。
- 当前生产池已有 20 条海克斯，尚未包含：
  - `18 / Courage of the Colossus / 巨像的勇气`
  - `19 / Dashing / 全凭身法`
  - `32 / Executioner / 裁决使`
- 这 3 条条目在事实层原始 JSON 中已存在，可确认基础字段：
  - `18`：prismatic，控制后获得护盾，原始描述仍带 `数值` 占位，需要人工还原与画像整理；
  - `19`：prismatic，位移类技能获得技能急速，原始描述仍带 `数值` 占位；
  - `32`：silver，低血增伤并在参与击杀后重置基础技能，原始描述仍带 `数值` 占位。
- 现有质量闸门明确要求：
  - 海克斯 `verified` 必须同时具备 `official-client / riot-support / community-mirror`；
  - `id` 必须在 `src/data/augments/allowlist.ts`；
  - 含 `数值` 或未还原模板文本不得进入 `verified`。
- 当前推荐回归脚本位于 `scripts/audit-recommendations.mjs`，适合继续追加定向场景，不需要改脚手架。
- 当前前端海克斯图标优先从 `src/data/icon-overrides.ts` 取本地映射，否则使用 profile 自带 `iconPath`；由于现有 `public/icons/augments` 中没有 `18/19/32` 对应本地资源，执行时需要做一个明确选择：
  - 若直接复用事实层 `iconPath` 可稳定展示，则不新增本地图标资源；
  - 若该路径在当前页面不可用，则需要补最小本地图标与映射。
- 当前工作区已存在未提交改动：
  - `scripts/audit-recommendations.mjs`
  - `src/data/champions/index.ts`
  - `src/data/icon-overrides.ts`
  - `src/data/champions/twelfth-batch.ts`
  - `src/data/champions/thirteenth-batch.ts`
  - 以及已有计划文档
  执行时必须避免覆盖与本任务无关的现有改动。

## Proposed Changes

### 1. 新增一个只包含这 3 条条目的 augment batch 文件

- 新建 `src/data/augments/seventh-batch.ts`
- 只录入：
  - `18 / Courage of the Colossus`
  - `19 / Dashing`
  - `32 / Executioner`
- 每条条目都需要补齐：
  - `id / name / localizedName / rarity / summary / officialDescription / iconPath`
  - `rewardTypes / triggerTypes / preferredUsers / playstyleFits / riskTags`
  - `positiveSignals / trapSignals`
  - `verification`
- 画像方向固定为：
  - `Executioner`
    - 归入 `damage`
    - 触发轴包含 `on_kill`
    - 画像偏 `resetter / attacker / caster` 中至少与当前规则可解释的一组对齐
    - 用于补齐残局收割与基础技能刷新轴
  - `Dashing`
    - 归入 `utility` 或 `damage+utility`，但以“位移技能急速”作为主轴
    - 触发方式以 `passive` 为主，偏好用户围绕 `dash / mobility / engage`
    - 用于补齐位移联动轴
  - `Courage of the Colossus`
    - 归入 `survival`
    - 触发轴包含 `on_cc`
    - 偏好用户对齐 `tank / melee`
    - 用于补齐控制转生存轴，与现有 `Slap Around` 形成对位关系
- 若原始事实层描述仍带占位，则执行时必须依据同条目可验证字段或可信镜像还原为可上线文本；若无法还原，则该条不得标 `verified`。

### 2. 接入生产海克斯索引

- 修改 `src/data/augments/index.ts`
- 引入 `seventhBatchAugmentProfiles`
- 按现有聚合方式追加到生产数组中，不调整无关 batch 的组织结构

### 3. 扩大生产 allowlist

- 修改 `src/data/augments/allowlist.ts`
- 仅新增 3 条 allowlist 记录：
  - `18 / Courage of the Colossus`
  - `19 / Dashing`
  - `32 / Executioner`
- 不修改已有 blocked 列表，不顺手调整其它候选范围

### 4. 处理海克斯图标的最小必需改动

- 修改范围只限与这 3 条图标直接相关的最小实现
- 首选方案：
  - 若 profile 中的 `iconPath` 在当前前端链路可直接展示，则只在数据文件中保留该路径，不补本地图标资源
- 兜底方案：
  - 若直接展示不可用，则最小补充 `public/icons/augments/18.webp`、`19.webp`、`32.webp`
  - 并在 `src/data/icon-overrides.ts` 中补 3 条映射
- 不为无关旧海克斯补图标，不改查询页逻辑

### 5. 补充推荐回归样例

- 修改 `scripts/audit-recommendations.mjs`
- 只增加与这 3 条海克斯直接相关的回归，不重写旧样例
- 建议最少新增以下方向：
  - `Executioner`：
    - 选择至少 1 个残局/收割/重置型英雄做正例
    - 目标是让它在旧的泛用伤害海克斯对照中能拿到合理第一
  - `Dashing`：
    - 选择至少 1 个明显依赖 dash/blink 的英雄做正例
    - 对照组应包含现有泛用输出或技能流海克斯，验证位移轴不是空转
  - `Courage of the Colossus`：
    - 选择至少 1 个坦控英雄，与 `Slap Around / Perseverance / Goredrink` 对照
    - 目标不是强行第一，而是确保它能成为合理前列且不被明显错配压掉
- 回归原则：
  - 若实测排名存在两个等价强解，可以用 `expectedTopAugmentIds`
  - 若某条新海克斯在当前规则下无法形成清晰、可解释、可回归的排序，则应停止放行并回到评审，不为了凑 3 条强行上线

### 6. 仅在必要时补最小规则支持

- 默认不改 `src/lib/recommendation/rules.ts`
- 只有在新增回归无法通过，且问题明确是“当前规则完全不认识 `on_kill / on_cc survival / dash haste` 轴”时，才允许加最小正向识别
- 若需要改规则，范围限定为：
  - 只为这 3 条海克斯及其对应轴补最少的 baseline / reward / risk 判断
  - 不重构整体评分系统，不顺手改旧 archetype

## Assumptions & Decisions

- 决策：这次只补 3 条明确指定海克斯，不同时补英雄，不扩大到其它候选。
- 决策：优先通过新增数据与回归完成接入，规则文件只有在必要时才动，符合“必须回归，不做多余修改”。
- 决策：新增海克斯 batch 文件采用 `src/data/augments/seventh-batch.ts`，因为当前生产海克斯 batch 只到 `sixth-batch.ts`。
- 决策：若任一条目的描述无法从当前可信来源中还原掉 `数值` 占位，则该条不应被标记为 `verified`，也不应进入 allowlist。
- 决策：执行时必须保护现有脏工作区，不覆盖与本任务无关的未提交改动。
- 假设：现有事实层与镜像源足够支持还原这 3 条条目的正式描述与基础画像；若发现证据不足，应以“停放该条”为先，而不是扩大修改面。

## Verification Steps

1. 核对三条海克斯的数据审计条件：
  - 证据源齐全
  - 描述无未还原占位
  - `id` 已在 allowlist
2. 运行 `npm run audit:data`
3. 运行 `npm run audit:recommendations`
4. 运行 `npm run lint`
5. 运行 `npm run build`
6. 抽样确认：
  - 查询页能看到这 3 条海克斯
  - 图标能正常显示，或在约定方案下稳定走 fallback
  - 新增回归样例能解释这 3 条海克斯的排序结果
