## Proposed Changes

### 1. 强化数据审计可扩性

**文件**

- [audit-data.mjs](file:///c:/Users/29403/Documents/trae_projects/hero-pro/scripts/audit-data.mjs)
- [data-audit.md](file:///c:/Users/29403/Documents/trae_projects/hero-pro/docs/data-audit.md)

**做什么**

- 把当前“手工枚举 batch 文件”的审计方式改成“按目录自动发现 `*-batch.ts` 文件并逐个加载”。
- 继续保留现有校验项：
  - 英雄 `id / key` 唯一性
  - 海克斯 `id / name` 唯一性
  - 核心字段完整性
  - `verified` 条目证据组合完整性
- 文档同步更新为“新增批次无需手工接入审计”的流程说明。

**为什么**

- 当前质量闸门已经有用，但仍依赖手工把新批次文件加进脚本。
- 批次继续增长后，这种方式容易漏检，和“数据底座质量优先”的目标冲突。

**怎么做**

- 在 `scripts/audit-data.mjs` 中读取：
  - `src/data/champions`
  - `src/data/augments`
- 只加载命名符合 `*-batch.ts` 的文件。
- 对每个文件中的导出数组执行与当前一致的审计逻辑。
- 审计输出继续保留 “Data audit passed: x champions, y augments” 的聚合结果。

### 2. 扩充第四批英雄画像

**文件**

- `src/data/champions/fourth-batch.ts`
- [champions/index.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/data/champions/index.ts)

**做什么**

- 新增第四批英雄，固定为：
  - `Ashe`
  - `Leona`
  - `Katarina`
- 每个英雄继续按现有结构补齐：

**怎么做**

- 使用现有同步脚本抓取原始资料：
  - `npm run sync:sources -- --champions=Ashe,Leona,Katarina`
- 仅使用 Riot `Data Dragon` 的 `zh_CN / en_US` 双语静态数据做事实来源。
- 中文技能名与称号来自 `zh_CN`，机制归纳来自 `en_US` 文本。
- 新增后在索引文件显式并入数组。

### 4. 给候选海克斯列表加滚动容器

**文件**

- [query-panel.tsx](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/components/query-panel.tsx)
- [globals.css](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/app/globals.css)

**做什么**

- 为候选海克斯区域增加独立可滚动容器。
- 保持单个海克斯按钮的当前样式和选中逻辑不变。
- 在桌面端限制候选区最大高度，在移动端允许更高但仍可滚动。

**为什么**

- 用户已经明确要求候选海克斯要支持滚动。
- 现有列表随着条目增长会把页面下半部分不断往下推，影响“快速做决策”的核心体验。<mccoremem id="03g0d7kwgycl1cm3c3qo4n8zd" />

**怎么做**

- 在 `query-panel.tsx` 中为 `.augment-list` 外层增加一个专门容器，例如 `.augment-scroll-area`。
- 在 `globals.css` 中设置：
  - `max-height`
  - `overflow-y: auto`
  - `padding-right`
  - 轻量滚动条样式
- 桌面端建议目标：
  - 最大高度约 `480px - 560px`
- 移动端建议目标：
  - 最大高度约 `50vh - 60vh`

### 5. 强化候选对比展示

**文件**

- [query-panel.tsx](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/components/query-panel.tsx)
- [globals.css](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/app/globals.css)
- [rules.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/lib/recommendation/rules.ts)
- [catalog.ts](file:///c:/Users/29403/Documents/trae_projects/hero-pro/src/lib/recommendation/catalog.ts)

**做什么**

- 把结果区从“排序列表”升级成“候选对比”：
  - 顶部继续显示当前最优候选
  - 每个候选显示与第一名的分差
  - 每个候选固定展示 2 条“主要推荐原因”与 1 条“主要风险提示”
- 对规则层补一小步精细化，让新扩数据的差异更容易被反映出来：
  - 支持 `heal_shield` 画像对辅助类海克斯加权
  - 支持 `resetter` 对滚雪球或收割类海克斯加权
  - 支持“长手功能射手”和“硬控前排”在基础画像上获得更明确的区别

**为什么**

- 现在用户能看到排序，但还不能直观看出“第二名为什么输给第一名”。
- 你的产品定位是“候选项即时推荐器”，不是百科页，所以需要强调相对选择理由。

**怎么做**

- 结果数据层不新增 API，继续复用 `recommendForAugmentCandidates()`。
- 在 UI 层计算：
  - `topScore`
  - `delta = topScore - currentScore`
- 在卡片里新增简洁文案，例如：
  - `比当前最优低 8 分`
  - `主要优势`
  - `主要风险`
- 保持页面文字简洁，不把全部规则拆给用户。
