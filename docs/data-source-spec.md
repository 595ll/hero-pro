# Hero Pro 数据来源规范

## 目标

- 保证英雄机制画像与海克斯符文画像的来源可追溯、可验证、可复查。
- 避免把主观经验、过时资料或社区评级误当作事实。
- 为后续推荐引擎提供统一的证据标准和入库流程。

## 数据来源分级

### A 级：权威直接来源

可直接作为事实入库，但仍需记录版本与抓取时间。

- Riot Developer Portal / Data Dragon
  - 用途：英雄列表、英雄名称、本地化文本、技能描述、图标、被动、基础元数据。
  - 说明：属于 Riot 面向第三方开发者提供的静态数据。
- Riot Support / Riot 官方模式说明
  - 用途：海克斯大乱斗（Arena）规则、强化符文通用机制、模式限制、稀有度、局内流程。
  - 说明：可用于确认“模式层面”的事实，例如强化符文存在银/金/棱彩三个层级。
- 游戏客户端内可验证的官方文本
  - 用途：强化符文正式名称、正式描述、图标、局内展示文案。
  - 说明：若后续接入客户端数据抓取或手动录入，必须保留截图或原始文本备份。

### A- 级：客户端导出文本镜像

在 Riot 没有公开静态接口、但存在客户端导出 JSON 的情况下，可作为生产可用来源组合的一部分。

- CommunityDragon `cdragon/arena/*.json`
  - 用途：Arena 强化符文的客户端导出名称、描述、tooltip、稀有度、图标、数值占位符与部分计算字段。
  - 说明：底层内容来自客户端导出，但传输载体不是 Riot 官方站点。
  - 边界：这是客户端原始导出池，不等于项目最终允许上线的生产海克斯池。
  - 使用前提：
    - 必须与 Riot 官方 Arena 模式说明交叉确认模式语义
    - 必须与强化符文条目清单交叉确认 `id / name / rarity / icon`
    - 必须记录具体数据版本

### B 级：高可信辅助来源

不可单独作为机制真值，必须与 A 级来源或客户端实测互相印证后才能入库。

- CommunityDragon 等客户端资源镜像
  - 用途：补齐图标、资源路径、部分未在 Data Dragon 暴露的客户端文本或资源索引。
  - 风险：并非 Riot 官方发布站点，存在结构变动和镜像时差。
- 高质量社区资料站
  - 用途：条目清单校对、中文名比对、图标映射、缺漏排查。
  - 风险：可能存在评级主观性、版本滞后、翻译差异。
- `arammayhem.com`
  - 用途：可参考海克斯符文中文名、图标、条目完整性。
  - 风险：站点已明确说明不受 Riot 认可，因此不能直接作为机制定论来源。

### C 级：经验与验证记录

只能用于推荐修正、争议标注和待验证项，不可直接写成机制事实。

- 高水平玩家经验总结
- 实战录像复盘
- 本地对局测试记录
- 人工推导结论

## 入库原则

### 1. 英雄机制画像

以下字段优先来自 A 级来源，若需要推导，必须保留推导依据：

- 英雄基础身份：英雄 ID、名称、别名、远近战
- 技能描述：被动、Q/W/E/R 文本
- 可见机制：位移、控制、护盾、治疗、召唤、持续区域、强化普攻
- 触发特征：高频施法、多段命中、击杀刷新、进场爆发、持续拉扯

以下字段允许建立“解释性标签”，但必须有证据：

- 主要输出来源
- 爆发 / 持续输出 / 消耗倾向
- 生存方式
- 海克斯适配标签

### 2. 海克斯符文画像

以下字段必须来自 A 级来源或 A + B 双重印证：

- 海克斯 ID
- 英文名 / 中文名
- 稀有度
- 正式描述
- 图标资源

对于 Arena 强化符文，若 Riot 没有公开静态接口，可接受以下等价验证组合：

- Riot 官方 Arena 模式说明
- 客户端导出文本镜像 `cdragon/arena/*.json`
- 条目清单镜像 `cherry-augments.json`

满足以上组合后，可将海克斯条目标记为 `verified`，前提是：

- 描述文本、稀有度、名称、图标路径彼此一致
- 数值占位符可直接读取或能从同条目数据字段还原
- 没有与其他来源冲突的显著异常
- 条目已进入项目维护的 Arena production allowlist

以下字段允许通过规则整理得到，但必须能解释：

- 触发方式
- 受益英雄特征
- 风险标签
- 泛用 / 特化属性
- 顺风 / 逆风适配倾向

### 3. 绝不直接入库的内容

- “这个海克斯某网站评 S 级，所以一定强”
- “这个英雄大家都这么玩”
- 未标版本的截图结论
- 无法复现或无法定位来源的社区说法
- 只凭个人感觉写出的英雄标签或海克斯标签

## 证据链要求

每条核心数据至少保留一个证据记录，推荐结构如下：

```ts
type EvidenceRecord = {
  sourceType:
    | "riot-ddragon"
    | "riot-support"
    | "official-client"
    | "community-mirror"
    | "community-site"
    | "manual-test";
  sourceName: string;
  sourceUrl?: string;
  version?: string;
  locale?: string;
  retrievedAt: string;
  confidence: "high" | "medium" | "low";
  note?: string;
};
```

规则：

- A 级来源默认可标 `high`
- B 级来源默认最多标 `medium`
- 仅人工测试或推导时默认标 `low` 或 `medium`
- 进入正式推荐逻辑的关键机制，整体置信度不得低于 `medium`

## 字段置信度策略

- `verified`
  - 有 A 级直接来源，或 A + B / A + 实测交叉验证
  - Arena 强化符文也可使用“Riot Arena 模式说明 + 客户端导出文本镜像 + 条目清单镜像”组合
- `provisional`
  - 有一定证据，但尚未达到正式上线标准
- `needs_review`
  - 信息来源冲突、版本不一致或仅有推导结论

未达到 `verified` 的条目不得参与正式生产推荐。

## 版本策略

- 每次数据抓取必须记录游戏版本或数据版本。
- 英雄与海克斯文本尽量对齐同一版本窗口。
- 版本变更后，先标记受影响条目，再做局部复核，不允许静默沿用旧结论。

## 推荐系统使用边界

- 推荐引擎只消费“已验证”的画像字段。
- 争议机制可以展示为“待验证提示”，但不计入主评分。
- 若某英雄或海克斯资料尚未达到上线标准，前台必须明确提示“暂未完整支持”。

## 合规与展示要求

- 网站需展示 Riot 要求的非官方声明。
- 不展示 Riot 明确不允许公开展示的 Arena 增强项胜率数据。
- 产品以辅助决策为目标，给出多个候选与原因，不做替用户强制决策的黑盒结论。

## 当前建议的数据源组合

英雄画像：

- 首选 Riot Data Dragon
- 辅助 Riot 客户端实测 / 官方文本

海克斯符文画像：

- 首选 Riot 官方模式说明 + 客户端官方文本
- 辅助 CommunityDragon 资源镜像
- 使用社区站点做图标、中文名与条目完整性校对

## 当前结论

- `arammayhem.com` 适合做参考与比对，不适合作为最终机制依据。
- 英雄基础资料已经有明确的权威静态数据入口。
- 海克斯全量资料需要走“官方说明 + 客户端文本 + 辅助镜像 + 人工校验”的组合方案，才能同时兼顾完整性与准确性。
- 本次复盘确认：`cdragon/arena/*.json` 只能作为原始候选池，不能直接等同于生产可收录池。
- 后续任何海克斯进入生产前，除证据链达标外，还必须先进入 `src/data/augments/allowlist.ts`。
